# Production Readiness Checklist

هذه القائمة تضبط الحد الأدنى المطلوب قبل إطلاق النسخة الأولية.

## ما تم تجهيزه داخل المستودع

- توحيد أوامر الفحص عبر `npm run check`
- إضافة `typecheck` و`check` إلى `package.json`
- إضافة `.env.example` ومنع تتبع ملفات البيئة محلياً
- إغلاق لوحة الإدارة افتراضياً من خلال `VITE_ENABLE_ADMIN`
- تقليل الحزمة الأولية عبر `lazy loading` للصفحات الثقيلة
- تجهيز GitHub Actions لتشغيل الفحوص على `push` و`pull_request`

## مطلوب قبل أول نشر

1. تأكيد قيم البيئة في منصة الاستضافة:
   - `VITE_SUPABASE_URL`
   - `VITE_SUPABASE_PUBLISHABLE_KEY`
   - `VITE_SITE_URL`
   - `VITE_APP_ENV=production`
   - `VITE_ENABLE_ADMIN=false` ما لم يتم إنهاء المصادقة
   - `VITE_MONITORING_ENABLED=true`
   - `VITE_MONITORING_ENDPOINT=<your-ingestion-endpoint>`
2. تطبيق جميع ترحيلات `Supabase` على بيئة الإنتاج.
3. مراجعة سياسات `RLS` في `Supabase` للمنتجات والفئات والملفات.
4. التأكد من وجود نسخ احتياطية وخطة rollback بسيطة.
5. ربط النطاق النهائي وضبط HTTPS.
6. التحقق من فهرسة SEO الأساسية وبيانات المشاركة الاجتماعية إذا كانت مطلوبة للإطلاق.
7. إضافة مراقبة أخطاء فعلية مثل Sentry أو Application Insights إذا كان المشروع سيدخل استخداماً حقيقياً.

## مخاطر معروفة حالياً

- لا توجد اختبارات وحدات أو اختبارات تكامل حتى الآن.
- المراقبة تعتمد على endpoint خارجي يجب ضبطه في متغيرات البيئة قبل الإطلاق.

## معيار القبول قبل Go-Live

- `npm run check` يمر محلياً وداخل CI
- البناء الإنتاجي ينجح بدون أخطاء
- الإدارة مغلقة أو محمية فعلياً
- endpoint المخصص لمراقبة الأخطاء يستقبل الأحداث بنجاح
- بيانات `Supabase` وسياساتها متوافقة مع سيناريو الإنتاج
